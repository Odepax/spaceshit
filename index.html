Spaceshit A temporary language to generate data structures

WTF is Spaceshit?

Recently, I have had to generate a lot of Java beans, applying JPA annotations on most of them. It is fair to say that this is a quite long and repetitive work to do, even using IDEs' getters & setters and method generation tools...

Thus, I thought it would be great to be able to generate class skeletons from a language-agnostic description of the data structure they represent. A single-use piece of code would be parsed and a complete class would be generated from it.

Syntax elements

person {
                id     -- positive
                name   -- !empty
  birth date  : calendar
  age         : int    -- computed
  best friend : person -- optional
  friends     : list of person
} .java -- bean !check !tostring

Types

The Spaceshit syntax supports generic types: use ... of ... to stack a generic. Multiple generics (e.g.: Map<K, V>) are not supported.

Like names, words in multi-words types are separated with spaces in order to be completely language-agnostic.

Names

Names are optional. If not explicitly specified, the name of a property will be inferred from the type. Generic types, such as list of pet, will be reversed to form the generated name of the property (e.g.: pet list).

In addition, Spaceshit defines some special types that will be converted to another type when used without a property name:

  • x, y and z are mapped to float;
  • id is mapped to int;
  • name and title are mapped to string.

Like types, words in multi-words names are separated with spaces in order to be completely language-agnostic.

Modifiers

Modifiers are space-separated flags that give extra information about a property or a whole structure. A modifier is either unspecified, present, or negated when prefixed with an exclamation mark.

As they are parsed, but not handled by the parser, modifiers may have different effects depending on the code generator used to generate actual code from a Spaceshit definition.

Target language

This gives the actual programming language to target when generating code.

Idioms

Here are some standards that should be respected when building a parser or a code generator in order to maintain a certain cohesion of the Spaceshit language.

The Spaceshit code is not meant to be saved

A Spaceshit structure definition is supposed to be a single-use code snippet, it should not be saved in a file and serve to generate code on a build-basis for instance.

Plus, if the technology changes, I want to be able to break retro-compatibility in order to change with it, and not be stuck nor forgotten...

The generated code is not meant to be final nor compilable

Keep in mind that the generated codes are class skeletons. As such, they may not be usable out of the box, and even require some tweaking before being compiled in some cases.

The generated code may not be well formated

... which should not be a problem as everyone reformat their code using the IDE, right?
